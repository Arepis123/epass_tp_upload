## 1. Overview

**ePass TP Upload** is a desktop application for Construction Labour Exchange Centre Berhad (CLAB) that automates the uploading of worker ePass and Green Card (GC/TP) PDF documents. For each worker it will:

- Read and extract information from PDF documents
- Rename the files using passport number, worker name, and years in Malaysia
- Upload the renamed files to an FTP server
- Notify the API to update the worker's record
- Send an email notification to the contractor
- Generate an Excel report of the entire run

---

## 2. System Requirements

- Windows 10 / 11
- The `ePassTPUpload` folder must remain intact — the `.exe` depends on all files inside it
- Network access to the FTP server, API, and both databases

---

## 3. Database Setup

The system connects to **two separate databases**.

### 3.1 Login Database (`[LoginDB]`)

Stores user accounts for application login.

```sql
CREATE TABLE `user` (
  `username` VARCHAR(100) NOT NULL,
  `password` VARCHAR(100) NOT NULL,
  `name`     VARCHAR(150) NOT NULL
);

INSERT INTO `user` VALUES ('admin', 'yourpassword', 'Administrator');
```

### 3.2 Main Database (`[DB]`)

The existing workers database. Two tables are referenced:

| Table | Column used | Purpose |
|---|---|---|
| `workers` | `wkr_passno`, `wkr_currentemp` | Fall-back contractor ID lookup |
| `contractors` | `ctr_clab_no`, `ctr_comp_name` | Company name lookup by CLAB ID |

## 4. Configuration (config.ini)

`config.ini` sits next to the `.exe`. It is created automatically with default values on first run. Fill in every section before use.

```ini
[LoginDB]
host     = 127.0.0.1
port     = 3306
database = your_login_database
username = root
password =

[DB]
host     = 127.0.0.1
port     = 3306
database = workers
username = root
password =

[FTP]
host       = ftp.example.com
port       = 21
username   = ftp_user
password   = ftp_password
remote_dir = /attach

[API]
base_url = https://example.com/eclabapi

[Schedule]
epass_folder =
gc_folder    =
run_time     = 08:00

[Email]
smtp_host        = smtp.gmail.com
smtp_port        = 587
username         =
password         =
sender_name      = ePass TP Upload
sender_email     =
use_tls          = true
bcc              =
test_recipient   =
```

### Key settings

| Setting | Description |
|---|---|
| `[LoginDB]` | Database containing the `user` login table |
| `[DB]` | Main workers database |
| `[FTP] remote_dir` | Root folder on the FTP server (e.g. `/attach`) |
| `[API] base_url` | Base URL of the CLAB API (no trailing slash) |
| `[Email] bcc` | If set, every outgoing email will silently BCC this address. Useful for audit/monitoring. |
| `[Email] test_recipient` | If set, **all** emails are redirected to this address instead of the real contractor — use for testing only. Clear when done. |

## 5. Login

When the application launches, a login window appears before the main interface.

- Credentials are validated against the `user` table in the `[LoginDB]` database
- The logged-in user's **name** (from the `name` column) is recorded in the Excel report under **Generated By**
- Closing the login window exits the application

## 7. Processing Workflow

When **Process & Upload** is clicked, the system performs these steps in order:

### Step 1 — Read PDFs

All `.pdf` files in the selected folders are read using `pdfplumber`.

**ePass fields extracted:**
- Passport No
- Worker Name
- Nationality
- VP No
- ePass Expiry
- **Year in Malaysia** (the number shown in the green triangle at the top right of the document, values 1–13)

**GC fields extracted:**
- Passport No
- Worker Name
- Nationality
- GC Expiry

If a passport number **cannot be extracted** from a PDF, that file is marked as **Failed** immediately and skipped — no further processing is attempted for it.

### Step 2 — Rename Files

Files are renamed immediately after extraction, **regardless of what happens later** (upload or API failure will not prevent renaming).

See [Section 8](#8-file-renaming-rules) for the naming format.

### Step 3 — Fetch Pending Worker List from API

The system calls `api_epass.php` to get the list of workers with pending documents. A document is considered pending (needs upload) if its value in the API response is `null`, `""`, or blank.

If the API is unavailable, the system falls back to the `workers` table in the `[DB]` database.

### Step 4 — FTP Upload

For each worker with pending documents, the system:

1. Creates the remote directory `{remote_dir}/{CLAB_ID}/` if it does not exist
2. Uploads the ePass PDF (if needed)
3. Uploads the GC PDF (if needed)

### Step 5 — API Notification

After upload, the system calls the update APIs:

- `api_epass_upd.php` — for ePass documents
- `api_tp_upd.php` — for GC documents

### Step 6 — Email Notification

On successful upload + API update, an HTML email is sent to the contractor. See [Section 11](#11-email-notifications).

### Step 7 — Archive & Report

Processed files are moved to the `Archive/` folder and an Excel report is generated. See [Sections 9](#9-excel-report) and [10](#10-archive).

## 8. File Renaming Rules

Files are renamed using the format:

```
{Passport No} - {Worker Name} - {Year}Y_{Document Type}.pdf
```

**Examples:**
```
10440224 - PULAMI BAL KUMAR - 8Y_ePass.pdf
10440224 - PULAMI BAL KUMAR - 8Y_GC.pdf
```

| Condition | Result |
|---|---|
| Name, passport, **and** year all readable | `{Passport} - {Name} - {Year}Y_{Type}.pdf` |
| Name and passport readable, year **not** found | `{Passport} - {Name}_{Type}.pdf` |
| Name **not** readable (`UNKNOWN`) | File is **not renamed** (kept as-is) |

**Name fallback:** If the name cannot be read from the PDF, the system attempts to use the `Worker_Name` from the API listing. Double spaces in names (e.g. `MD  OKIL`) are automatically normalised to single spaces.

## 9. Excel Report

The report is saved to the `Report/` folder as `EPassReport_{timestamp}.xlsx` and has two sheets.

### Sheet 1 — Run Summary

| Row | Description |
|---|---|
| Generated | Date and time the report was created |
| Generated By | Name of the logged-in user (or `Server` for scheduled runs) |
| Elapsed Time | Total processing time in seconds |
| Archive Folder | Archive subfolder name for this run |
| Total Workers | Number of unique workers processed |
| Success | Workers fully uploaded and API updated |
| Already Complete | Workers skipped because all documents were already set |
| Upload Failed | Workers where FTP upload failed |
| API Failed | Workers uploaded but API update failed |
| Skipped (Not in DB) | Workers whose passport was not found in the database |
| Skipped (No CLAB ID) | Workers with no CLAB ID returned |
| Read Errors | PDF files that could not be parsed |
| ePass / GC / TP | Count of each document type uploaded |
| Email stats | Applications emailed, failed, or skipped |

### Sheet 2 — Upload Summary

One row per worker with these columns:

`#` · `Passport No` · `Worker Name` · `Nationality` · `Contractor ID` · `Company` · `Year in MY` · `ePass` · `GC` · `Status` · `App No` · `CLAB ID` · `Timestamp` · `ePass API` · `GC API` · `Email` · `Error`

**Company** — fetched from the `contractors` table using the worker's CLAB ID (`ctr_clab_no` → `ctr_comp_name`).

**Year in MY** — years the worker has been in Malaysia, extracted from the ePass PDF.

**Row colours:**
- Green — Success
- Red — Failed / Error
- Grey — Skipped

## 10. Archive

After each run, processed PDFs are moved to:

```
Archive/
└── {YYYYMMDD_HHMMSS}/
    ├── Done/
    │   ├── ePass/    ← successfully uploaded ePass docs
    │   └── GC/       ← successfully uploaded GC docs
    └── Failed/       ← only created if there are failures
        ├── ePass/
        └── GC/
```

- `Done/` — workers with status: **Success** or **Skipped - Already complete**
- `Failed/` — workers with any other status (including **Uploaded / API Failed**), plus any PDF files that could not be parsed
- The `Failed/` folder is **not created** if all files processed successfully

---

## 11. Email Notifications

One email is sent per unique **Application Number** (`App_No`) to the contractor's email address returned by the API.

The email contains an HTML table listing all workers under that application, with a tick (✓) for each document uploaded or "Already set" if it was previously uploaded.

### Testing Emails

To redirect all emails to yourself during testing, set `test_recipient` in `config.ini`:

```ini
[Email]
test_recipient = yourname@example.com
```

The log will show `[TEST MODE] All emails will be redirected to: yourname@example.com`.

Clear `test_recipient` when done testing to restore normal delivery.

## 12. Scheduler (Automated Daily Run)

Click **Schedule...** in the main window to open the scheduler dialog.

### Setting up a scheduled task

1. Set the **Run time** (HH:MM, 24-hour format)
2. Browse to the **ePass folder** and **GC folder**
3. Click **Register Task** — this creates a Windows Task Scheduler entry

The task runs the application with a `--scheduled` flag, which:
- Skips the login window entirely
- Processes all PDFs in the configured folders
- Sends email notifications
- Generates a report saved to `Report/`
- Logs everything to `Log/upload.log`
- Shows **"Server"** as the Generated By in the report

### Managing the task

| Button | Action |
|---|---|
| Register Task | Create or update the scheduled task |
| Disable / Enable Task | Temporarily pause or resume the schedule |
| Remove Task | Delete the scheduled task permanently |

### Run History

The scheduler dialog shows a history of all previous scheduled runs parsed from `Log/upload.log`, including worker counts and success/failure totals.

## 13. Troubleshooting

| Problem | Likely cause | Fix |
|---|---|---|
| Login fails with "Database Error" | `[LoginDB]` settings incorrect or DB unreachable | Check host, port, database, username, password in `config.ini` |
| "Config.ini not found" on launch | `config.ini` missing next to the `.exe` | Run the app once to auto-generate it, then fill in the values |
| Year in Malaysia not extracted | PDF layout differs from expected format | The year will be absent from the filename and report; the file will still be renamed using name and passport only |
| Worker skipped — "Already complete" | API shows documents already uploaded AND not blank | If the document should be re-uploaded, clear the value in the API/database first |
| Worker skipped — "Not in DB" | Passport number not in the `workers` table | Ensure the worker is registered in the system |
| FTP upload fails | Wrong FTP credentials or network issue | Verify `[FTP]` settings; check firewall / VPN |
| Emails not sent | SMTP credentials missing or incorrect | Fill in the `[Email]` section in `config.ini` |
| Files not renamed | Worker name could not be read from PDF | Check the PDF is a valid ePass/GC document; review the log for read errors |
| `Failed/` folder not created | All files processed successfully | This is expected behaviour — `Failed/` only appears when there are failures |
